    • Celery architecture deep dive

    Published on Sunday, February 6, 2022
    Tags: celery

    The Celery project describes itself as:

    Celery is a simple, flexible, and reliable distributed system to process vast amounts of messages , while providing operations with the tools required to maintain such a system.

    It’s a task queue with focus on real-time processing, while also supporting task scheduling.

    And the documentation goes into great detail about how to configure Celery with its plethora of options, but it does not focus much on the high level architecture or how messages pass between the components. Celery is extremely flexible (almost every component can be easily replaced!) but this can make it hard to understand. [1]

    The traditional simplified view of a Celery application.

    Celery has a few main components:

    1. Your application code, including any Task objects you’ve defined.
    2. A broker or message transport.
    3. One or more celery workers.
    4. A (results) backend.

    Let’s break this down a bit more and look into how these components communicate.

    Note

    The following assumes you’re using the standard “prefork” configuration of a celery worker (although the explanation doesn’t change too much if you’re using gevent, eventlet, or threads).

    The broker and celery worker broken into sub-components

    Application code

    My experience using Celery was mostly in a mixed Django & Twisted application, but for this example a simple Python script will be used. You define a Celery a Celery application (which includes a bunch of configuration of how to connect to the broker and backend) and then use the task decorator, e.g.:

    from celery import Celery

app = Celery(
    "example",
    backend="rpc://",
    broker="amqp://guest@localhost//",
)

@app.task()
def add(a, b):
    return a + b

    Your application can then request that the tasks be run and wait for the results: [2]

    from my_app import add

# Request that the tasks run and capture their async results.
task_1 = add.delay(1, 2)
task_2 = add.delay(3, 4)

result_1 = task_1.get()
result_2 = task_2.get()
print(f"Results: {result_1}, {result_2})

    Broker

    The message broker is some other application which takes the task requests and queues them until a worker is ready to process them. Common options include RabbitMQ and Redis.

    As the diagram above shows the broker itself has some sub-components, including an exchange and one or more queues. How those interact (and how you might configure them) is not what I’m interested in presenting here. The Celery routing documentation has more information.

    Workers

    Celery workers fetch queued tasks from the message brokers and then run the code inside of your task, they can optionally return a result to provide to the results backend.

    Celery workers have a “main” process which fetches tasks from the broker, by default it requests many tasks at once: the “prefetch multiplier x concurrency“. (If your prefetch multiplier is 5 and your concurrency is 4, it attempts to fetch up to 20 queued tasks from the broker.) Once it has these in memory it deserializes them and runs whatever their Strategy is — for a normal Celery Task this essentially feeds them to each worker in the processing pool individually.

    Backend

    A key-value storage for the results of running tasks. These can be queried by using the AsyncResult object which is returned to your application code.
